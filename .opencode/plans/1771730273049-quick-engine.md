# Inbound CLI + Mailbox Wrapper Plan

## Goal
- Build a new CLI under `packages/` that covers the E2 API surface.
- Add a mailbox wrapper that persists default mailbox settings in `inbound.json` and applies mailbox-scoped defaults to mail search/list commands, with explicit address/domain overrides.

## What We Will Build

### 1) New package: `packages/inbound-cli`
- Create a Bun-first TypeScript CLI package with `bin` entrypoint `inbound`.
- Keep architecture layered:
  - command parser + routing
  - API adapter layer
  - mailbox/config layer
  - output/error layer

Planned file layout:
- `packages/inbound-cli/package.json`
- `packages/inbound-cli/tsconfig.json`
- `packages/inbound-cli/src/bin/inbound.ts`
- `packages/inbound-cli/src/cli.ts`
- `packages/inbound-cli/src/commands/**`
- `packages/inbound-cli/src/client/**`
- `packages/inbound-cli/src/mailbox/**`
- `packages/inbound-cli/src/output/**`
- `packages/inbound-cli/tests/**`

### 2) E2 command coverage (resource-aligned)
Use command groups that mirror `app/api/e2`:
- `inbound domains list|get|create|update|delete`
- `inbound addresses list|get|create|update|delete` (alias for `email-addresses`)
- `inbound emails list|get|send|update|cancel|reply|retry`
- `inbound endpoints list|get|create|update|delete|test`
- `inbound attachments get`
- `inbound mail threads list|get`
- `inbound guard list|get|create|update|delete|check|generate`

### 3) API access strategy
- Reuse `inboundemail` SDK for existing resources it already supports:
  - `attachments`, `domains`, `emailAddresses`, `emails`, `endpoints`, `mail`
  - from `packages/inbound-typescript-sdk/src/resources/*`
- Add thin direct E2 HTTP wrappers for routes missing from SDK:
  - `guard/*`
- Exclude onboarding commands from CLI v1 (confirmed) because those routes currently rely on session-auth patterns instead of the API-key auth used by the rest of E2.
- Standardize global flags:
  - `--api-key`, `--base-url`, `--json`, `--debug`
- Config precedence:
  - CLI flags > env vars (`INBOUND_API_KEY`, `INBOUND_BASE_URL`) > `inbound.json` > defaults

### 4) Mailbox wrapper (`inbound.json`)
Introduce local config + mailbox selection.

Proposed initial schema:
```json
{
  "version": 1,
  "api": {
    "baseUrl": "https://inbound.new"
  },
  "mailbox": {
    "default": "primary",
    "mailboxes": {
      "primary": {
        "name": "Primary",
        "email": "support@example.com",
        "addresses": ["support@example.com"],
        "domains": ["example.com"]
      }
    }
  }
}
```

Mailbox commands:
- `inbound mailbox init`
- `inbound mailbox list`
- `inbound mailbox add <key> --name --email [--address] [--domain]`
- `inbound mailbox use <key>`
- `inbound mailbox remove <key>`
- `inbound mailbox show`

Config discovery:
- `--config <path>` override first.
- Otherwise discover nearest `inbound.json` by walking upward from current working directory.
- No home-directory fallback in v1.

Mailbox resolution behavior:
- For mailbox-aware list/search commands (`emails list`, `mail threads list`):
  1. Resolve active mailbox from `--mailbox` or config default.
  2. Start with mailbox-configured addresses/domains.
  3. If explicit `--address`/`--domain` is passed, replace mailbox defaults for that run.
  4. Normalize + dedupe values.
  5. Map to E2 query params.
- Support repeated `--address` and `--domain` flags in CLI; since E2 endpoints take single values, perform request fan-out and merge/dedupe results client-side.

### 5) Output and error model
- Human-readable output by default; strict payload output with `--json`.
- Preserve API error context while improving actionability:
  - validation/usage errors with fix hints
  - auth errors with clear token/env guidance
  - network errors with retry-friendly messages

## Implementation Phases

### Phase A - Scaffold + core runtime
- Create package scaffold and bin entry.
- Add global option parsing and command registry.
- Implement API client factory with env + flag config.

### Phase B - Mailbox/config foundation
- Implement `inbound.json` read/validate/write utilities.
- Implement mailbox CRUD commands.
- Implement mailbox resolution and command-context injection.

### Phase C - Core E2 commands
- Implement resource groups backed by SDK where available:
  - domains, addresses, emails, endpoints, attachments, mail threads

### Phase D - Remaining E2 commands
- Implement guard commands via direct E2 wrappers.
- Document onboarding exclusion in help/docs for v1 scope clarity.

### Phase E - Hardening
- Add help polish and command examples.
- Add tests for config resolution, mailbox behavior, command-to-request mapping, and merge behavior for multi address/domain filters.
