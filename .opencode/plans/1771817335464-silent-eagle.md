# Plan: Abuse-Response Tooling via Admin API

## Goal
Build an operator-safe abuse-response flow that:
- audits offenders with concise output (no SQL by default),
- performs immediate enforcement through **admin API**,
- reads `.env` for `INBOUND_API_KEY`,
- supports default cascade action: **suspend tenant + ban user + block domains**.

## Scope
1. Add missing admin API endpoints for suspend/ban/unban.
2. Add a CLI/tooling wrapper that calls admin API (dry-run by default).
3. Update slash-command UX to use concise operator flow and action mode.
4. Keep SQL appendix opt-in only.

## Implementation Plan

### 1) Add missing admin APIs (backend)

#### 1.1 Tenant suspend endpoint
- Create: `app/api/e2/admin/tenants/suspend.ts`
- Route: `POST /admin/tenants/:tenantId/suspend`
- Auth: `validateAdminAndRateLimit` (`app/api/e2/admin/lib/auth.ts`)
- Action: call `suspendTenantSending` from `lib/aws-ses/aws-ses-tenants.ts`
- Behavior:
  - idempotent success if already suspended,
  - if AWS/config set unavailable, return structured error,
  - keep response schema status-code keyed.

#### 1.2 User ban endpoint
- Create: `app/api/e2/admin/users/ban.ts`
- Route: `POST /admin/users/:userId/ban`
- Auth: `validateAdminAndRateLimit`
- DB: `user` table (`lib/db/auth-schema.ts`)
- Request body: optional `reason`, optional `banExpires` (ISO)
- Behavior:
  - set `banned=true`, `banReason`, `banExpires`, `updatedAt`.

#### 1.3 User unban endpoint
- Create: `app/api/e2/admin/users/unban.ts`
- Route: `POST /admin/users/:userId/unban`
- Auth: `validateAdminAndRateLimit`
- Behavior:
  - set `banned=false`, clear `banReason` + `banExpires`, update timestamp.

#### 1.4 Register routes
- Update: `app/api/e2/[[...slugs]]/route.ts`
- Add `.use(suspendTenant)`, `.use(banUser)`, `.use(unbanUser)`.

### 2) Build admin-API abuse action wrapper

#### 2.1 New script/tool wrapper
- Create: `scripts/admin-abuse-block.ts`
- Add package script in `package.json`:
  - `"admin:abuse-block": "bun run scripts/admin-abuse-block.ts"`

#### 2.2 Auth + config
- Load env via `dotenv`.
- Require `INBOUND_API_KEY`.
- Base URL default: `https://inbound.new/api/e2` (override via env/flag).
- All calls use `Authorization: Bearer ${INBOUND_API_KEY}`.

#### 2.3 CLI contract
- Inputs:
  - `--tenant-id <id>` (required)
  - `--user-id <id>` (optional; resolve from admin tenant lookup if missing)
  - `--domains <csv>` (optional; resolve from tenant/user if missing)
  - `--reason "..."` (required for execute)
  - `--execute` (absent => dry-run)
  - `--confirm <token>` (required for execute)
  - `--json` (optional)

#### 2.4 Cascade flow (default)
- Resolve target IDs.
- Planned actions:
  1) suspend tenant (`POST /admin/tenants/:tenantId/suspend`)
  2) ban user (`POST /admin/users/:userId/ban`)
  3) block each domain (`POST /admin/blocked-signup-domains`)
- Domain block `409` should be treated as idempotent/already blocked.

#### 2.5 Safety controls
- Dry-run by default (no mutations).
- Confirmation token required for execute.
- Return structured action results per step (`success`, `status`, `error`).

### 3) Command UX improvements

#### 3.1 Keep `/inbound` concise by default
- Update `/.opencode/commands/inbound.md` to:
  - default windows: 7d + 30d,
  - concise sections only,
  - SQL only when `--include-sql` is present,
  - action instructions use `bun run admin:abuse-block ...`.

#### 3.2 Add focused enforcement command
- Create `/.opencode/commands/abuse-block.md`:
  - purpose: run dry-run or execute cascade for specified tenant/user/domains,
  - examples for dry-run and execute,
  - output contract focused on action summary and IDs.

### 4) Optional phase-2 custom OpenCode tool
- Create `/.opencode/tools/abuse-actions.ts` as a thin wrapper around `admin:abuse-block` once workflow is stable.
- Keep command-driven flow first for lower integration risk.

## Verification Plan

### API verification
1. Run targeted tests for new admin routes (auth + happy path + idempotency + not found).
2. Validate route registration via E2 endpoint calls.

### Tool verification
1. Dry-run:
   - `bun run admin:abuse-block --tenant-id <id> --reason "incident" --json`
   - confirm no mutations, shows resolved targets + planned steps.
2. Execute:
   - `bun run admin:abuse-block --tenant-id <id> --reason "incident" --execute --confirm <token> --json`
   - verify tenant suspended, user banned, domains blocked.

### End-to-end operational verification
1. Run `/inbound` without args: concise 7d+30d summary and candidate IDs, no SQL dump.
2. Run `/inbound --include-sql`: SQL appendix included.
3. Run `/abuse-block` dry-run then execute path to confirm operator ergonomics and safety gates.
